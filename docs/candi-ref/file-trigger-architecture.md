# ファイルトリガー型アーキテクチャ

作成日: 2026-05-15

## 概要

「チャットに3つのエージェントを常駐させ、必要なタイミングでプロンプトをファイル生成し、そのファイルをトリガーにAIにタスクを実行させる」というアーキテクチャの実現可能性を調査する。

## 結論

**技術的に実現可能。** 複数のプロジェクトで同様の仕組みが実用化されている。

---

## 実現方式

### 方式A: ファイル監視型（推奨）

```text
[ユーザーがチャットで指示]
       │
       ▼
[オーケストレーターがプロンプトファイル生成]
       │
       ▼
[File Watcher が変更を検知]
       │
       ▼
[3つのエージェントに並行タスク配信]
       ├──► スキラー（MiMo-V2.5-Pro）
       ├──► ホーク（DeepSeek V3.2 TEE）
       └──► キーパー（Kimi K2.5 TEE）
       │
       ▼
[各エージェントが結果ファイルを生成]
       │
       ▼
[File Watcher が変更を検知]
       │
       ▼
[オーケストレーターが結果を統合・表示]
```

**実績のあるプロジェクト:**
- **OpenLegion**: `watchers` でディレクトリ監視、エージェント起動
- **Osaurus**: 200ms〜10minの段階的応答設定
- **initrunner**: `file_watch` トリガー、globパターン、debounce対応
- **AI4PKM**: Obsidian vaultの変更を検知→複数CLIエージェントにルーティング

### 方式B: Webhook型

```text
[外部イベント（GitHub, Slack, etc.）]
       │
       ▼
[Webhook受信]
       │
       ▼
[オーケストレーター]
       │
       ▼
[エージェント起動]
```

**実績のあるプロジェクト:**
- **relayfile**: Webhook→仮想ファイルシステム→エージェント読み取り
- **Fast.io**: ファイルアップロード→Webhook→エージェント実行

### 方式C: ハイブリッド型

ファイル監視とWebhookを組み合わせる。

- ローカル開発: ファイル監視
- CI/CD: Webhook
- 手動トリガー: チャット入力

## 技術詳細

### ファイル監視の実装

```python
# Python: watchfiles (initrunner/Osaurusで実績)
from watchfiles import awatch

async for changes in awatch('./prompts'):
    for change_type, path in changes:
        if path.endswith('.md'):
            prompt = read_file(path)
            await dispatch_to_agents(prompt)
```

```javascript
// Node.js: chokidar
const chokidar = require('chokidar');
const watcher = chokidar.watch('./prompts/*.md');
watcher.on('add', (path) => {
    const prompt = fs.readFileSync(path, 'utf-8');
    dispatchToAgents(prompt);
});
```

### プロンプトファイルの形式

```markdown
# prompts/2026-05-15_design-review.md

## タスク
設計書のレビューを3つの視点から実施する。

## 対象ファイル
- docs/spec/architecture.md

## 指示
各エージェントは以下の観点からレビューしてください:
- スキラー: 技術最適化と堅牢性
- ホーク: ユーザ視点と要件拡充
- キーパー: セキュリティ、法務、経理視点

## 出力形式
Markdown形式で、各立場の意見を構造化して出力してください。
```

### 結果ファイルの形式

```markdown

## 討論結果

[意見]

### ホーク（DeepSeek V3.2 TEE）
[意見]

### キーパー（Kimi K2.5 TEE）
[意見]

### 統合判断
[オーケストレーターによる統合]
```

## 「それ以上」にできること

### 1. エージェント間の直接対話

ファイルを介さず、エージェント間で直接メッセージを交換できる。

```text
スキラー: 「この設計には技術的な問題がある」
ホーク: 「ユーザー体験としてどの部分が問題になるか？」
キーパー: 「事故時の影響範囲も確認したい」
```

- **AutoGen**: エージェント間の会話をネイティブにサポート
- **LangGraph**: グラフベースのメッセージフロー

### 2. 動的エージェント生成

タスクの複雑さに応じて、エージェント数を動的に増減できる。

- **HCP-MAD**: 簡単なタスクは2エージェントで早期停止、複雑なタスクは追加エージェントで拡張
- **CascadeDebate**: 不確実性が高い場合のみマルチエージェント審議を起動

### 3. 外部ツール統合

エージェントが外部ツール（コード実行、Web検索、データベース検索）を呼び出せる。

- **DeepSeek**: Function Calling対応
- **Kimi**: 300ステップのtool calling対応
- **GPT**: Responses API + MCP対応

### 4. 永続メモリ

エージェントが過去の討論を記憶し、蓄積された知識で判断を改善できる。

- **DCI**: 共有ワークスペースでの構造化された集合思考
- **DOVA**: 永続ユーザーモデル + エンティティ認識コンテキスト

### 5. 人間-in-the-loop

討論の途中で人間が介入し、方向性を修正できる。

- **CascadeDebate**: 最終フォールバックとして人間を配置
- **DCI**: 再開条件付きの少数派報告を生成
