# 利用シーン候補

作成日: 2026-05-15

## ユーザー提示の利用シーン

### 1. 設計書の要件洗練

3つの視点から設計書の要件を洗練させる。

- スキラー: 「この要件は将来の拡張を阻害する」
- ホーク: 「この要件はユーザーにとって本当に必要か？」
- キーパー: 「この要件は事故・法務・コスト面で許容できるか？」

### 2. 実装結果の評価

3つの視点から実装結果を評価する。

- スキラー: 「コードの構造は保守性が高いか？」
- ホーク: 「ユーザーに届けるにはこれで十分か？」
- キーパー: 「テストが通っていても、事故時の影響は許容できるか？」

---

## 追加提案の利用シーン

### 3. 技術選定の討論

特定の技術選定について、複数の視点から議論する。

- 例: 「monorepo vs polyrepo」「REST vs gRPC」「RDB vs NoSQL」
- 各エージェントが異なる技術スタックの経験を持つ前提で議論

### 4. API設計のレビュー

APIのインターフェース設計について、複数の視点からレビューする。

- スキラー: 「APIの一貫性と将来性」
- ホーク: 「開発者体験（DX）」
- キーパー: 「エラーケース、認可、監査ログ」

### 5. PR（Pull Request）のレビュー

コード変更について、3つの視点からレビューする。

- スキラー: 「アーキテクチャへの影響」
- ホーク: 「この変更でユーザーの問題が解決されるか？」
- キーパー: 「セキュリティリスク、コスト、運用事故」

### 6. 要件の矛盾検出

複数の要件ドキュメント間の矛盾を検出する。

- 各エージェントが異なる要件ドキュメントを読み、整合性を議論
- 隠れた前提条件の発見に有効

### 7. リスク分析

特定の設計や実装方針について、リスクを多角的に分析する。

- スキラー: 「技術的に何が起きるか？」
- ホーク: 「ユーザーへの影響は何か？」
- キーパー: 「そのリスクの法務・費用・運用影響は？」

### 8. 障害復旧シナリオの検討

障害発生時の復旧シナリオを、複数の視点から検討する。

- 各エージェントが異なる障害パターンを想定
- 対応手順の網羅性が向上する

| API設計のレビュー | ★★★★ | ★★★★ | ★★★ |
| PRレビュー | ★★★★ | ★★★★★ | ★★★★ |
| 要件の矛盾検出 | ★★★★★ | ★★★ | ★★★★ |
| リスク分析 | ★★★★ | ★★★★ | ★★★ |
| 障害復旧シナリオ | ★★★ | ★★★ | ★★★ |

## 推奨

初期は **「設計書の要件洗練」** と **「実装結果の評価」** に集中し、discussion protocol を確立してから他のシーンに展開するのが合理的。
